# Gym Management System

End-to-end local setup for the **Gym API (Laravel + MySQL + Passport)** and **Gym UI (Angular)**.

## Tech Stack

### Frontend
- Angular (v21) + Angular Material
- Tailwind CSS
- ApexCharts (`apexcharts`, `ng-apexcharts`)

### Backend
- Laravel (v12) + PHP (8.2+; project uses 8.4 in your environment)
- Laravel Passport (OAuth2 / token auth)
- MySQL (database)
- CORS enabled for `api/*`

## Project Structure
- `gym-api/` : Laravel backend
- `gym-UI/` : Angular frontend

## Dashboards
- **SuperAdmin**: `/admin/*` (analytics, owners, gyms, notifications/broadcast tools)
- **Owner**: `/owner/*` (overview, members, staff, products, nutrition, revenue, etc.)
- **Nutritionist**: `/nutritionist/*`
  - `dashboard`: KPIs (clients, my plans, active/expiring plans, unread notifications)
  - `clients`: member list scoped to nutritionist’s gyms (from `/users`)
  - `nutrition-plans`: create/update/delete **own** plans (from `/nutrition-plans`) with gym + client assignment

### Nutritionist RBAC/Scoping Notes
- `id_nutritionist` is always derived from the authenticated user in UI for create/update requests.
- UI only exposes edit/delete actions for plans owned by the logged-in nutritionist.
- UI applies defensive filtering (member-only clients and own plans) even if an endpoint returns broader data.
- Gym context is header-driven (`X-Gym-Id`) via interceptor; nutritionist gym options are fetched from `/gyms`.
- If no gym is assigned, nutrition plan creation is blocked with an explicit empty-state/error message.

### Frontend Route Structure
- Nutritionist area is lazy-loaded from `gym-UI/src/app/features/nutritionist/nutritionist.routes.ts`.
- Main entries:
  - `/nutritionist/dashboard`
  - `/nutritionist/clients`
  - `/nutritionist/nutrition-plans`

## Installation

### 1) Backend (`gym-api`)
From repo root:
1. Install PHP deps:
   - `cd gym-api`
   - `composer install`
2. Install backend frontend assets (Vite/Tailwind tooling inside `gym-api`):
   - `npm install`
3. Configure environment:
   - Copy env file: `copy .env.example .env`
   - Set MySQL + app URL in `gym-api/.env`:
     - `DB_CONNECTION=mysql`
     - `DB_HOST=127.0.0.1`
     - `DB_PORT=3306`
     - `DB_DATABASE=<your_database_name>`
     - `DB_USERNAME=<your_mysql_user>`
     - `DB_PASSWORD=<your_mysql_password>`
     - `APP_URL=http://localhost:8000` **(or** `http://127.0.0.1:8000` on systems where `localhost` doesn’t resolve in PHP** )**
4. Generate key:
   - `php artisan key:generate`
5. Run migrations and seed:
   - `php artisan migrate --force`
   - `php artisan passport:install --force --no-interaction`
   - `php artisan db:seed --force`

### 2) Frontend (`gym-UI`)
1. Install node deps:
   - `cd ../gym-UI`
   - `npm ci --legacy-peer-deps`
2. (Optional) If you prefer, you can use `npm install --legacy-peer-deps`.

## Running the Project

### Backend
The recommended working command in this environment is:
- `cd gym-api`
- `php -S 127.0.0.1:8000 -t public public/index.php`

Then open:
- Health check: `http://127.0.0.1:8000/up`

### Frontend
- `cd gym-UI`
- `ng serve --port 4200`

Then open:
- `http://localhost:4200/`

### Ports
- Backend API: `8000`
- Frontend dev server: `4200`

## Database Setup

This project uses Laravel migrations + seeders.

Run:
- `php artisan migrate --force`
- `php artisan db:seed --force`

Passport tables/clients:
- Created via `php artisan passport:install --force --no-interaction`
- Seeders also call `passport:client` to create the personal access client.

## Troubleshooting

### `php artisan serve` fails (or prints “Failed to listen … reason: ?”)
Use the working backend command instead:
- `php -S 127.0.0.1:8000 -t public public/index.php`

### MySQL access errors during `php artisan migrate`
Common causes:
- Wrong `DB_USERNAME` / `DB_PASSWORD`
- Wrong `DB_DATABASE`

### Angular install fails with peer dependency conflicts
Use:
- `npm ci --legacy-peer-deps`

### Seeding fails with `Call to undefined function ... fake()`
This is caused by factories using Faker. The repo was fixed to use `$this->faker`.
Also ensure you ran:
- `composer install` (not only `--no-dev`)

### Passport install “table already exists” errors
If you previously ran Passport install attempts on the same DB, duplicates can appear.
Fix options:
1. Delete/clean duplicated Passport migration files created by failed runs, then re-run:
   - `php artisan passport:install --force --no-interaction`
2. Or (advanced): run `php artisan passport:keys --force` and ensure the personal client exists (seeders will create it).
